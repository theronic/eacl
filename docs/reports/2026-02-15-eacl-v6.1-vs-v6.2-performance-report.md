# EACL v6.1 vs v6.2 Comparative Performance Report

**Date:** 2026-02-15
**Branch:** `eacl/v6.2`
**Datomic:** In-memory (`datomic:mem://electric-eacl`)
**JVM:** temurin-jre-21
**Methodology:** Each version tested in a separate JVM with freshly seeded in-memory Datomic. 200-iteration warmup before timed runs. All benchmarks run via `future` threads with `System/nanoTime` precision.

## Versions Under Test

| | v6.1 | v6.2 |
|---|---|---|
| **EACL dep** | `~/code/eacl` | `~/code/eacl-cursor-tree` |
| **Branch** | `feature/write-schema-dsl` | `feature/cursor-tree` |
| **Cursor format** | v1: `{:resource {:id eid}}` | v2: `{:v 2, :e eid, :p {path-idx min-val}}` |
| **Key difference** | Single cursor EID shared across all paths | Per-path state tracking (cursor-tree) |
| **DB servers** | 372,715 | 373,037 |

## Dataset

- **500 accounts**, each with 10-15 users and 500-1000 servers
- **~373k total servers**
- **~6,250 users** (500 accounts x ~12.5 avg)
- **3 fixture users** (user-1, user-2, super-user)
- **1 platform** with super-user as super_admin
- Permission model: `user->owner->account->account->server` (2-hop arrow permission)

## Results

### 1. `can?` (permission check, 3-arg arity)

| Metric | v6.1 | v6.2 | Delta |
|--------|-----:|-----:|------:|
| Median | 0.018 ms | 0.018 ms | **0%** |
| Min | 0.013 ms | 0.013 ms | 0% |
| Avg | 0.030 ms | 0.027 ms | -10% |
| n=500, warmup=100 | | | |

**Verdict: No meaningful difference.** Both versions use the same indexed `can?` implementation. The cursor-tree changes don't affect point permission checks.

> **Bug found:** The map arity `(can? acl {:subject ... :resource ...})` throws `AssertionError` because `consistency` defaults to nil instead of `fully-consistent`. Use the 3-arg arity `(can? acl subject permission resource)` as a workaround. Reported at `core.clj:252`.

| 10 | 9.01 ms | 8.34 ms | -7% |
| 50 | 12.68 ms | 8.27 ms | **-35%** |
| 100 | 9.04 ms | 9.68 ms | +7% |

| 1,000 | 10.18 ms | 20.15 ms | **+98%** |

**Verdict: Mixed.** v6.2 is faster at low limits (10-50) but shows a regression at high limits (1000+). The cursor-tree's per-path state management adds overhead that becomes visible when materializing large result sets in a single call.

The v6.1 `lookup-resources` scales more gracefully with limit because it uses a simpler cursor format with less bookkeeping. The v6.2 cursor-tree tracks volatile state per permission path, which adds overhead proportional to result count.

**Verdict: No significant improvement in steady-state pagination.**

Both versions achieve ~7-13 ms per page across all depths (pages 50-500). The v6.1 cursor format `{:resource {:id last-eid}}` already provides efficient cursor-based seeking because each permission path uses `d/index-range` starting from the cursor EID.

The v6.2 cursor-tree's theoretical advantage (per-path state tracking) does not produce measurable improvement in this workload because:
1. The v6.1 simple cursor already provides O(limit) per-page work via `d/index-range` cursor seeking
2. The permission graph for this dataset has a simple structure (2-hop arrow: `account->admin + shared_admin`)
3. The cursor-tree overhead (volatile state tracking, per-path min tracking) roughly cancels out any benefit

The one clear win for v6.2 is **cold-start page 0: 8.83 ms vs 94.62 ms** (-91%). This suggests the v6.2 warmup path is more efficient, likely because cursor-tree initialization is lighter than v6.1's first-page setup.

## Analysis

### Why cursor-tree doesn't show expected improvement

The cursor-tree optimization targets a specific scenario: **multiple permission paths with divergent intermediate traversals**. The v2 cursor `{:v 2, :e eid, :p {path-idx min-intermediate-eid}}` tracks per-path state so that each path can resume from its own position rather than re-scanning from a shared cursor EID.

In this benchmark's permission model (`user->owner->account->account->server`), the permission graph is relatively uniform:
- All paths through the arrow permission traverse the same kind of intermediate (accounts)
- The single cursor EID in v6.1 is sufficient because the index range scan via `d/index-range` naturally seeks to the right position
- The v6.1 cursor already provides O(limit) per-page behavior for this graph structure

The cursor-tree would show more benefit with:
- **Many divergent permission paths** (e.g., multiple arrow permissions through different intermediate types)
- **Highly skewed intermediate distributions** where some paths produce many results and others few
- **Very deep pagination** where per-path state prevents redundant intermediate lookups

### Overhead profile of v6.2

v6.2 introduces:
- Volatile state tracking per path (`vswap!` on `!min`, `!state`)
- V2 cursor construction with path state map
- `extract-cursor-eid` and `build-v2-cursor` helper overhead
- Per-result `tracking-min` bookkeeping

This adds ~15-20 us per call for fast operations (small result sets, `lookup-subjects`) and becomes more noticeable at high limits.

## Conclusion

For this workload (2-hop arrow permission, ~373k servers, in-memory Datomic), **v6.1 and v6.2 perform nearly identically for pagination**, with v6.1 having a slight edge on raw throughput and v6.2 winning on cold-start latency. The cursor-tree optimization does not produce the expected 20-100% improvement because v6.1's simpler cursor already provides efficient cursor-based index seeking for this permission graph structure.

The cursor-tree's value proposition may be more apparent with complex permission graphs having multiple divergent arrow paths, which this benchmark does not exercise.
