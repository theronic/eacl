Why EACL?

Built for production Datomic applications that need sophisticated access control

Situated Authorization

Permission data lives next to your application data in Datomic. No network hops to external auth services. Query permissions with the same connection you use for everything else.

Full Consistency

Unlike eventually consistent systems, EACL queries the same Datomic database as your application. Permission changes are immediately reflected. No sync delays or stale checks.

Graph Traversal

Define complex permission hierarchies with arrows. Users who own an account automatically get admin on all servers in that account. EACL traverses the graph for you.

SpiceDB Compatible

Familiar concepts if you've used Zanzibar, SpiceDB, or similar systems. Relations, permissions, and relationships work the same way. Easy migration path.

Pure Clojure

No external dependencies. No gRPC. No protobuf. Just clean Clojure code that runs in your JVM. Easy to debug, easy to extend, easy to understand.

Battle Tested

Used in production at CloudAfrica. Handles real-world multi-tenant access control for cloud infrastructure. Proven at scale with complex permission hierarchies.

Simple, Expressive API

Define schemas and check permissions with clean Clojure code

eacl_example.clj 
;; Define relations between resource types and subjects
(Relation :account :owner :user)
(Relation :account :platform :platform)
(Relation :server :account :account)
(Relation :vpc :account :account)

;; Define permissions - direct from relations
(Permission :account :admin {:relation :owner})

;; Permissions via arrows - traverse the graph!
(Permission :server :admin
  {:arrow :account :permission :admin})

(Permission :vpc :admin
  {:arrow :account :permission :admin})

;; Chain permissions - :view implies :admin
(Permission :server :view {:permission :admin})
;; Create relationships between entities
(eacl/create-relationships! acl
  [(->Relationship
     (->user "alice")
     :owner
     (->account "acme-corp"))

   (->Relationship
     (->account "acme-corp")
     :account
     (->server "prod-web-1"))

   (->Relationship
     (->account "acme-corp")
     :account
     (->vpc "main-vpc"))])

;; Now alice has :admin on acme-corp
;; And :admin on prod-web-1 (via arrow)
;; And :admin on main-vpc (via arrow)
;; Check if a user has a permission
(eacl/can? acl
  (->user "alice")
  :admin
  (->server "prod-web-1"))
;; => true

;; Find all resources a user can access
(eacl/lookup-resources acl
  {:subject       (->user "alice")
   :permission    :admin
   :resource/type :server
   :limit         100})
;; => {:data [{:type :server :id "prod-web-1"}]
;;     :cursor ...}

;; Find all users who can access a resource
(eacl/lookup-subjects acl
  {:resource     (->server "prod-web-1")
   :permission   :admin
   :subject/type :user
   :limit        100})
;; => {:data [{:type :user :id "alice"}]
;;     :cursor ...}

Get Started in Minutes

Add EACL to your Datomic project and start securing your resources

1

Add the Dependency

Add EACL to your deps.edn

io.github.theronic/eacl {:git/sha "LATEST"}
2

Install the Schema

Transact the EACL schema into your Datomic database

(require '[eacl.datomic.schema :as schema])
@(d/transact conn schema/v6-schema)
3

Create a Client

Initialize the EACL client with your connection

(require '[eacl.datomic.core])
(def acl (eacl.datomic.core/make-client conn {}))
4

Define & Check Permissions

Create your schema, add relationships, and start checking permissions

(eacl/can? acl subject permission resource)
